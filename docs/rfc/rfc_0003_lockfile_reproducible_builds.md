# RFC-0003: Lockfile & Reproducible Builds

## Status
Draft

## Summary
This RFC defines the **lockfile model** for `cmod`, ensuring **deterministic, reproducible C++ module builds** across machines, CI systems, and time.

The design is inspired by Go modules and Cargo, adapted to C++ realities:
- Multiple compilers
- Compiler- and flag-specific BMIs
- Source-based dependency distribution

---

## Motivation

Without a lockfile, C++ builds suffer from:
- Non-deterministic dependency resolution
- Silent upgrades via Git branches
- Irreproducible CI failures
- Undebuggable module cache mismatches

`cmod.lock` provides a **single source of truth** for dependency resolution.

---

## Design Goals

1. Deterministic builds from a clean checkout
2. Human-readable and auditable
3. Source-of-truth = Git commit hash
4. Safe for vendoring and air-gapped builds
5. Independent of compiler artifacts (PCMs)

## Non-Goals

- Locking ABI compatibility
- Sharing precompiled modules
- Replacing compiler caches

---

## Lockfile Overview

- Filename: `cmod.lock`
- Generated by: `cmod resolve`, `cmod build`, `cmod add`
- Never edited manually (but readable)
- Must be checked into version control

---

## Lockfile Structure

```toml
version = 1

[[module]]
name = "github.fmtlib.fmt"
source = "https://github.com/fmtlib/fmt"
commit = "a1b2c3d4e5f6"
tag = "v10.2.1"
checksum = "sha256:..."

[[module]]
name = "com.satyavis.math.core"
source = "https://git.satyavis.com/math/core"
commit = "deadbeef"
checksum = "sha256:..."
```

---

## Resolution Rules

### Rule 1: Commits Are Canonical

- Every dependency resolves to a **specific commit hash**
- Branch names are resolved once, then frozen
- Tags are recorded for audit only

---

### Rule 2: Pseudo-Versions

When no semantic tag exists, `cmod` generates Go-style pseudo-versions:

```
v0.0.0-20260128-deadbeef
```

Used for:
- Nightly dependencies
- Internal modules
- Experimental branches

---

### Rule 3: Minimal Version Selection (MVS)

`cmod` uses **Minimal Version Selection**, identical in principle to Go:

- For each module path, select the highest required version
- No SAT solver
- Deterministic and fast

Conflicts are resolved by upgrading, never downgrading.

---

## Update Semantics

### Explicit Updates Only

```bash
cmod update github.fmtlib.fmt
```

Rules:
- No implicit updates during `build`
- Lockfile changes are explicit and reviewable

---

## Vendoring

### Vendor Directory

```text
vendor/
└── github.fmtlib.fmt/
    └── src/
```

```bash
cmod vendor
```

Rules:
- Vendor contents are derived from `cmod.lock`
- Vendored builds ignore network access
- Vendored sources are immutable

---

## Offline & Hermetic Builds

When `vendor/` is present:

- Network access is forbidden
- Only vendored sources are used
- Lockfile must match vendor content

Violations are hard errors.

---

## Compiler Independence

The lockfile:
- Does NOT contain compiler versions
- Does NOT reference PCMs/BMIs
- Is stable across platforms

Compiler-specific artifacts live in the **local build cache**.

---

## Security Model

Each module entry includes:
- Git commit hash
- Content checksum

This prevents:
- Dependency confusion
- Tampering after fetch
- Silent history rewrites

Optional:
- Signed commits
- Signed tags

---

## Failure Modes

| Scenario | Behavior |
|--------|----------|
| Missing lockfile | Error (unless `cmod init`) |
| Dirty lockfile | Build refused |
| Vendor mismatch | Error |
| Network access in vendored mode | Error |

## Open Questions

- Should lockfile format be extensible?
- Should signed lockfiles be supported?
- How to handle mono-repos with multiple root modules?

---

## Conclusion

Reproducibility is non-negotiable for C++ modules.

`cmod.lock` provides a deterministic, auditable, and enterprise-safe foundation for building C++ module graphs—without pretending ABI problems don’t exist.
